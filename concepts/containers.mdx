NanoClaw runs all agents inside containers (lightweight Linux VMs) to provide true OS-level isolation. This is the primary security boundary that makes Bash access and code execution safe.

## Why containers?

Containers provide:

- **Process isolation** - Agent processes can't affect the host system
- **Filesystem isolation** - Agents only see explicitly mounted directories
- **Resource limits** - CPU/memory can be constrained (future)
- **Non-root execution** - Agents run as unprivileged user

<Info>
NanoClaw uses Docker by default for cross-platform compatibility. On macOS, you can use Apple Container instead (via `/convert-to-apple-container` skill).
</Info>

## Container architecture

### Base image

The container is built from `container/Dockerfile`:

```dockerfile
FROM node:22-slim

# Build arguments
ARG INSTALL_CJK_FONTS=false
ARG CLAUDE_CODE_VERSION=2.1.116
ARG AGENT_BROWSER_VERSION=latest
ARG VERCEL_VERSION=latest
ARG BUN_VERSION=1.3.12

# System dependencies (Chromium, fonts, tini, git, curl, unzip)
RUN apt-get update && apt-get install -y --no-install-recommends \
    chromium fonts-liberation fonts-noto-color-emoji \
    libgbm1 libnss3 ca-certificates curl git tini unzip \
    && rm -rf /var/lib/apt/lists/*

# Bun runtime (for agent-runner)
RUN curl -fsSL https://bun.sh/install | bash -s "bun-v${BUN_VERSION}"

# Agent-runner deps (source is NOT baked in — mounted RO at runtime)
WORKDIR /app
COPY agent-runner/package.json agent-runner/bun.lock ./
RUN bun install --frozen-lockfile

# Global Node CLIs via pnpm (layered for cache efficiency)
RUN corepack enable
RUN pnpm install -g "vercel@${VERCEL_VERSION}"
RUN pnpm install -g "agent-browser@${AGENT_BROWSER_VERSION}"
RUN pnpm install -g "@anthropic-ai/claude-code@${CLAUDE_CODE_VERSION}"

# Entrypoint with tini for PID 1 signal forwarding
COPY entrypoint.sh /app/entrypoint.sh

RUN mkdir -p /workspace/group /workspace/extra && \
    chown -R node:node /workspace
USER node
WORKDIR /workspace/group
ENTRYPOINT ["/usr/bin/tini", "--", "/app/entrypoint.sh"]
```

**Key components:**

- **Base**: `node:22-slim` (Debian-based, minimal)
- **Browser**: Chromium with all dependencies
- **Agent runtime**: Bun (runs TypeScript directly, no build step)
- **Tools**: `agent-browser`, `vercel`, `curl`, `git`
- **Claude SDK**: `@anthropic-ai/claude-code` (installed globally via pnpm)
- **PID 1**: `tini` for correct signal forwarding so DB writes finalize on shutdown
- **User**: `node` (uid 1000, non-root)
- **Working directory**: `/workspace/group`
- **CJK fonts**: Opt-in via `INSTALL_CJK_FONTS=true` build arg (saves ~200MB by default)

### Entrypoint flow

The entrypoint uses `tini` as PID 1 for signal handling, then runs the agent-runner directly with Bun:

1. **`tini`** handles PID 1 responsibilities (zombie reaping, signal forwarding)
2. **`entrypoint.sh`** runs `exec bun run /app/src/index.ts`
3. **Agent-runner polls** `inbound.db` for pending messages and writes results to `outbound.db`
4. **Container exits** when the poll loop has no work and the idle timeout expires, cleaned up via `--rm` flag

<Note>
The agent-runner source is a shared read-only bind mount from `container/agent-runner/src/` at `/app/src`. All groups share the same code — source-only changes never require an image rebuild. There is no per-group copy or recompilation step.
</Note>

## Volume mounts

Containers only see what's explicitly mounted. All groups use the same mount structure — there is no main/non-main distinction at the mount level.

### Standard mounts

| Host path | Container path | Mode | Purpose |
|-----------|---------------|------|---------|
| `data/v2-sessions/{group-id}/{session-id}/` | `/workspace` | Read-write | Session directory (inbound.db, outbound.db, .heartbeat, outbox/) |
| `groups/{folder}/` | `/workspace/agent` | Read-write | Agent group folder (CLAUDE.md, container.json, working files) |
| `groups/{folder}/container.json` | `/workspace/agent/container.json` | Read-only | Nested RO mount prevents agent from modifying its own config |
| `groups/{folder}/CLAUDE.md` | `/workspace/agent/CLAUDE.md` | Read-only | Composed CLAUDE.md (regenerated on every spawn) |
| `groups/global/` | `/workspace/global` | Read-only | Shared global memory |
| `container/CLAUDE.md` | `/app/CLAUDE.md` | Read-only | Shared base CLAUDE.md |
| `data/v2-sessions/{group-id}/.claude-shared/` | `/home/node/.claude` | Read-write | Claude SDK state, settings, skill symlinks |
| `container/agent-runner/src/` | `/app/src` | Read-only | Shared agent-runner source |
| `container/skills/` | `/app/skills` | Read-only | Shared skills (symlinked from `.claude-shared/skills/`) |

<Note>
Agent-runner source is shared read-only across all groups. There is no per-group copy — the same code runs everywhere. Source changes take effect on next container spawn without an image rebuild.
</Note>

### Mount security

All mounts are validated against the allowlist at `~/.config/nanoclaw/mount-allowlist.json`:

```json
{
  "allowedRoots": [
    {
      "path": "~/projects",
      "allowReadWrite": true,
      "description": "Development projects"
    }
  ],
  "blockedPatterns": ["password", "secret", "token"],
  "nonMainReadOnly": true
}
```

**Validation steps:**

1. Resolve symlinks to real path
2. Check against blocked patterns (`.ssh`, `.env`, etc.)
3. Verify path is under an allowed root
4. Enforce `nonMainReadOnly` for non-main groups
5. Enforce per-root `allowReadWrite` setting
6. Reject container paths with `..` or absolute paths

## Container lifecycle

### Spawn

```typescript
const container = spawn(CONTAINER_RUNTIME_BIN, [
  'run',
  '--rm',                  // Remove on exit
  '--name', containerName, // nanoclaw-v2-{group}-{timestamp}
  '-e', `TZ=${TIMEZONE}`,  // Pass timezone
  '--user', `${uid}:${gid}`, // Run as host user (if not root/1000)
  '-v', 'host:container',  // Volume mounts
  '--entrypoint', 'bash',  // Override default entrypoint
  imageTag,                // Base or per-group custom image
  '-c', 'exec bun run /app/src/index.ts'
]);
```

**User mapping:**

- If host uid is 0 (root): No `--user` flag set; Dockerfile default (`node`, uid 1000) applies
- If host uid is 1000: No `--user` flag set (already matches container default)
- If `process.getuid` is unavailable (native Windows without WSL): No `--user` flag set
- Otherwise: Run as host uid/gid (via `--user` flag), with `HOME=/home/node` explicitly set

This ensures bind-mounted files are accessible (same uid on both sides).

### Execute

1. **Agent-runner starts**: Bun runs the agent-runner TypeScript directly (no build step)
2. **Poll loop begins**: Container polls `inbound.db` for pending messages
3. **Messages processed**: Agent-runner invokes Claude SDK for each message
4. **Results written**: Responses written to `outbound.db` for host delivery
5. **Heartbeat maintained**: Container touches `/workspace/.heartbeat` for liveness detection
6. **Stderr logging**: SDK debug output logged at debug level

### Cleanup

Containers are ephemeral:

- **`--rm` flag**: Docker removes container on exit
- **Session DBs persisted**: `inbound.db` and `outbound.db` survive container death
- **Claude state persisted**: `.claude-shared/` mounted, survives restart
- **Group files persisted**: `/workspace/agent` mounted, survives restart

<Info>
Even if the container crashes, all data in mounted directories persists. Only the container itself is ephemeral. The host sweep detects crashed containers via stale heartbeat and can wake a replacement.
</Info>

## IO model

All communication between host and container goes through the session databases — there are no stdout markers, no stdin piping, and no IPC file watchers.

**Inbound (host → container):** The host writes `messages_in` rows to `inbound.db`. The container's poll loop detects new rows and processes them.

**Outbound (container → host):** The agent writes `messages_out` rows to `outbound.db`. The host's delivery poll detects new rows and delivers them through channel adapters.

**Liveness:** The container touches `/workspace/.heartbeat` periodically. The host sweep checks the file's mtime to detect stale or crashed containers.

This design eliminates the complexity of stdout parsing and IPC file coordination, and ensures exactly one writer per database file (no cross-mount lock contention).

## Stale detection

There is no host-side idle timeout or hard timeout in v2. Instead, stale and stuck containers are detected by the host sweep (`src/host-sweep.ts`), which runs every 60 seconds:

- **Heartbeat staleness**: Container touches `/workspace/.heartbeat` during operation. If the mtime goes stale beyond a threshold, the host considers the container stuck.
- **Processing claim age**: The `processing_ack` table in `inbound.db` tracks which messages the container is actively processing. Claims that remain outstanding too long indicate a stuck agent.
- **Container state**: The `container_state` table in `outbound.db` records the current tool-in-flight state, helping distinguish legitimate long-running work from truly stuck containers.

When the host sweep detects a stuck container, it kills it via `docker stop` and marks the session as stopped. The next inbound message will wake a fresh container.

<Note>
This approach avoids killing containers doing legitimate long-running work (like complex code generation or web scraping). Wall-clock timeouts are replaced by liveness-based detection.
</Note>

## Resource limits

Currently no CPU/memory limits enforced. Future enhancement:

```typescript
args.push('--memory', '2g');      // 2GB RAM limit
args.push('--cpus', '2');         // 2 CPU cores
args.push('--pids-limit', '100'); // Max 100 processes
```

## Logging

Container stderr is captured by the host process and logged at debug level. Container lifecycle events (spawn, exit, errors) are logged to `logs/nanoclaw.log`.

Host logs: `logs/nanoclaw.log` (normal) and `logs/nanoclaw.error.log` (errors only — some delivery/approval failures only show up here).

## Container runtime

NanoClaw uses a constant runtime binary (`docker` by default). To switch to Apple Container on macOS, use the `/convert-to-apple-container` skill.

**Supported runtimes:**

- **Docker** (default): Cross-platform, well-tested
- **Apple Container** (macOS): Lightweight, native virtualization

## Skills and MCP servers

Skills are managed via symlinks rather than bulk copying:

1. **Shared source**: Skills live in `container/skills/` and are mounted read-only at `/app/skills/` inside containers
2. **Per-group selection**: `container.json` controls which skills are enabled (`skills: 'all'` or an explicit list)
3. **Symlinks**: On every container spawn, the host syncs symlinks in `.claude-shared/skills/` to point to `/app/skills/{name}`
4. **Available to agent**: Claude Agent SDK discovers skills via the `.claude/skills/` directory

### Built-in container skills

| Skill | Description |
|-------|-------------|
| `/agent-browser` | Browse the web, fill forms, extract data |
| `/self-customize` | Agent self-modification (install packages, add MCP servers) |
| `/slack-formatting` | Format messages for Slack using mrkdwn syntax |
| `/welcome` | First-run welcome and setup guidance |

### Additional directory auto-discovery

Directories mounted at `/workspace/extra/*` (via additional mounts) are automatically passed to the SDK as `additionalDirectories`. This means any `CLAUDE.md` file in those directories is loaded automatically, giving the agent context about the mounted project.

### MCP servers

- Configured in agent-runner source (`container/agent-runner/src/index.ts`)
- Built-in: `nanoclaw` MCP server with tools organized into modules:
  - **Core**: `send_message`, `send_file`, `edit_message`, `add_reaction`
  - **Scheduling**: `schedule_task`, `list_tasks`, `cancel_task`, `pause_task`, `resume_task`
  - **Agents**: agent-to-agent communication tools
  - **Interactive**: question cards and approval requests
  - **Self-mod**: `install_packages`, `add_mcp_server` (require admin approval)
- Custom MCP servers can be added via `container.json`
- MCP server runs via Bun directly (no build step): `bun run /app/src/mcp-tools/index.ts`

<Accordion title="Adding a custom MCP server">
Add the MCP server to `container.json` in the agent group's folder:
```json
{
  "mcpServers": {
    "my-custom-server": {
      "command": "node",
      "args": ["/path/to/custom-server.js"],
      "env": {}
    }
  }
}
```
The agent-runner discovers additional MCP servers from `container.json` at startup. No image rebuild is needed — the configuration is read from the mounted file.
</Accordion>

## Per-agent-group images

Agents can request custom packages via the `install_packages` MCP tool (requires admin approval). When packages are installed, a custom Docker image is built on top of the base image:

```dockerfile
FROM nanoclaw-agent:latest
USER root
RUN apt-get update && apt-get install -y {apt-packages}
RUN pnpm install -g {npm-packages}
USER node
```

The resulting image is tagged as `nanoclaw-agent:{agent-group-id}` and stored in `container.json` as `imageTag`. Subsequent container spawns for that group use the custom image.

## Browser automation

Chromium runs inside the container:

- **Executable**: `/usr/bin/chromium`
- **CLI**: `agent-browser` (installed globally)
- **Headless**: Always (no display in container)
- **User data**: Stored in group folder (persists across runs)
- **Network**: Full access (same as host, no restrictions)

## Security implications

### What containers protect against

- **Filesystem access**: Agents can't read `~/.ssh` or other sensitive paths
- **Process interference**: Agents can't kill host processes or inject code
- **Persistence**: Containers are ephemeral, no state survives unless mounted
- **Privilege escalation**: Non-root execution limits kernel attack surface

### What containers DON'T protect against

- **Network access**: Agents have full network access (can exfiltrate data)
- **Mounted directory tampering**: Agents can modify anything in mounted directories
- **Vault-based API access**: Containers can make authenticated API requests through the secret injection layer (though they cannot extract real credentials)
- **Resource exhaustion**: No CPU/memory limits (can DoS host)

<Warning>
Containers provide filesystem isolation, not network isolation. Agents can make arbitrary HTTP requests and exfiltrate data over the network.
</Warning>

## Troubleshooting

### Container won't start

1. Check Docker is running: `docker ps`
2. Check image exists: `docker images | grep nanoclaw-agent`
3. Rebuild image: `./container/build.sh`
4. Check host logs: `logs/nanoclaw.log`

### Container appears stuck

1. Check `.heartbeat` mtime in the session directory
2. Look for long-running `processing_ack` claims in `inbound.db`
3. The host sweep should detect and kill truly stuck containers
4. Manually stop: `docker stop nanoclaw-v2-{group}-{timestamp}`

### Permission errors

1. Check mount paths are readable by host user
2. Check uid/gid mapping in container arguments
3. Verify allowlist includes path (for additional mounts)
4. Check symlink resolution didn't change path

### Messages not delivered

1. Check `outbound.db` for pending `messages_out` rows
2. Verify delivery polls are running (check host logs)
3. Confirm channel adapter is connected
4. Check `inbound.db` for the message status
