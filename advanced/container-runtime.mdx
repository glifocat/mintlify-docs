NanoClaw runs agents in isolated Linux containers to provide security through OS-level process and filesystem isolation. The container runtime is abstracted to support multiple backends.

## Runtime abstraction

All runtime-specific logic lives in `src/container-runtime.ts`, making it easy to swap runtimes. Currently supported:

- **Docker** (default) - Cross-platform support (macOS, Linux, Windows via WSL2)
- **Apple Container** (macOS only) - Lightweight native runtime

The runtime binary is specified by `CONTAINER_RUNTIME_BIN` in `src/container-runtime.ts`:

```typescript
/** The container runtime binary name. */
export const CONTAINER_RUNTIME_BIN = 'docker';
```

## Apple Container vs Docker

Apple Container is Apple's native virtualization framework (macOS 15+). It runs Linux containers without a VM layer like Docker Desktop, making it lighter weight and faster to start.

### When to use Apple Container

- You're on **macOS 15 (Sequoia) or later**
- You want to **avoid installing Docker Desktop** (and its licensing)
- You want **faster container startup** — no VM boot overhead
- You're running NanoClaw on a personal Mac, not a shared server

### When to stick with Docker

- You're on **Linux or Windows (WSL2)** — Apple Container is macOS-only
- You need **cross-platform parity** — Docker behaves identically across OS
- You're deploying to a **production server** (Linux VPS, Hetzner, etc.)
- You rely on Docker-specific tooling (Docker Compose, Portainer, etc.)

### Key differences

| | Docker | Apple Container |
|---|---|---|
| Binary | `docker` | `container` |
| Bind mounts | `-v host:container:ro` | `--mount type=bind,source=...,target=...,readonly` |
| Stop command | `docker stop -t 1 name` | `container stop name` |
| Health check | `docker info` | `container system status` |
| Orphan cleanup | Parses `docker ps --format` JSON | Parses `container ls --format json` |
| Host gateway | Built-in on macOS, `--add-host` on Linux | Built-in (`host.docker.internal`) |
| Platform | macOS, Linux, Windows (WSL2) | macOS 15+ only |

### Switching runtimes

Run the `/convert-to-apple-container` skill in Claude Code:

```
/convert-to-apple-container
```

The skill modifies `src/container-runtime.ts` to change the binary from `docker` to `container` and adjusts mount syntax, stop commands, and health check commands accordingly. The skill also configures credential proxy network binding (Phase 3) — see below. To revert, use `git revert`.

<Warning>
Apple Container uses the same `container/Dockerfile` to build the agent image. If you switch runtimes, rebuild the image with `container build` instead of `docker build`.
</Warning>

### Credential proxy network binding

Apple Container uses a bridge network (`bridge100`) that only exists while containers are running. The credential proxy must start before any container, so it cannot bind to the bridge IP — it must bind to `0.0.0.0` via `CREDENTIAL_PROXY_HOST=0.0.0.0` in `.env`. This exposes port 3001 on all network interfaces.

On shared or public networks, add a macOS `pf` firewall rule to block external access:

```bash
echo "block in on en0 proto tcp to any port 3001" | sudo pfctl -ef -
```

The `/convert-to-apple-container` skill handles this configuration automatically (Phase 3), including persisting the firewall rule to `/etc/pf.conf`.

<Warning>
Without the firewall rule, anyone on your local network can route API requests through the credential proxy using your credentials. On private/home networks this is typically acceptable; on public networks, always enable the firewall rule.
</Warning>

## Container image

The agent container is built from `container/Dockerfile` and includes:

- **Node.js 22** - Base runtime for globally-installed CLIs
- **Bun** - Runtime for the agent-runner (runs TypeScript directly, no build step)
- **Chromium** - Browser automation via agent-browser
- **Claude Code SDK** - Installed globally via pnpm as `@anthropic-ai/claude-code`
- **System tools** - `curl`, `git`, `tini`, `unzip`
- **System fonts** - Emoji support (CJK fonts opt-in via build arg)

### Dockerfile breakdown

<Accordion title="Base image and system dependencies">
  ```dockerfile
  FROM node:22-slim
  ARG INSTALL_CJK_FONTS=false

  RUN apt-get update && apt-get install -y --no-install-recommends \
      chromium fonts-liberation fonts-noto-color-emoji \
      libgbm1 libnss3 ca-certificates curl git tini unzip \
      && if [ "$INSTALL_CJK_FONTS" = "true" ]; then \
          apt-get install -y --no-install-recommends fonts-noto-cjk; \
         fi \
      && rm -rf /var/lib/apt/lists/*
  ```
  
  CJK fonts (~200MB) are opt-in via the `INSTALL_CJK_FONTS` build arg. `tini` handles PID 1 signal forwarding.
</Accordion>

<Accordion title="Bun runtime and agent-runner deps">
  ```dockerfile
  ARG BUN_VERSION=1.3.12
  RUN curl -fsSL https://bun.sh/install | bash -s "bun-v${BUN_VERSION}"

  WORKDIR /app
  COPY agent-runner/package.json agent-runner/bun.lock ./
  RUN bun install --frozen-lockfile
  ```
  
  Only dependencies are baked in. Agent-runner source is provided via a shared read-only bind mount at runtime — source changes never require an image rebuild.
</Accordion>

<Accordion title="Global Node CLIs via pnpm">
  ```dockerfile
  RUN corepack enable
  RUN pnpm install -g "vercel@${VERCEL_VERSION}"
  RUN pnpm install -g "agent-browser@${AGENT_BROWSER_VERSION}"
  RUN pnpm install -g "@anthropic-ai/claude-code@${CLAUDE_CODE_VERSION}"
  ```
  
  Layered for cache efficiency — bumping `claude-code` (the most common change) only invalidates one layer. `only-built-dependencies` gates pnpm's supply-chain policy.
</Accordion>

<Accordion title="Entrypoint">
  ```dockerfile
  COPY entrypoint.sh /app/entrypoint.sh
  RUN chmod +x /app/entrypoint.sh

  RUN mkdir -p /workspace/group /workspace/extra && \
      chown -R node:node /workspace
  USER node
  WORKDIR /workspace/group

  ENTRYPOINT ["/usr/bin/tini", "--", "/app/entrypoint.sh"]
  ```
  
  `tini` is PID 1, reaps zombies, and forwards signals cleanly. `entrypoint.sh` runs `exec bun run /app/src/index.ts` so Bun is tini's direct child.
</Accordion>

### Building the image

Rebuild the container image after modifying the Dockerfile or agent-runner:

```bash
./container/build.sh
```

<Warning>
The container buildkit caches the build context aggressively. `--no-cache` alone does NOT invalidate COPY steps. To force a clean rebuild, prune the builder:

```bash
docker builder prune -af
./container/build.sh
```
</Warning>

## Container lifecycle

### Spawning containers

Containers are spawned by the `wakeContainer` function in `src/container-runner.ts`, which deduplicates concurrent wake calls:

```typescript
export function wakeContainer(session: Session): Promise<void>
```

<Steps>
  <Step title="Resolve agent group and config">
    The host reads the agent group from the central DB and loads `container.json` for container configuration, provider, and package settings.
  </Step>
  
  <Step title="Prepare session state">
    Writes destinations, session routing, and runtime identity fields. Syncs skill symlinks and composes CLAUDE.md from the shared base + fragments.
  </Step>
  
  <Step title="Build volume mounts">
    The host determines which directories to mount based on the agent group's configuration and any provider-contributed mounts.
  </Step>
  
  <Step title="Generate container name">
    Each container gets a unique name: `nanoclaw-v2-{group}-{timestamp}`
  </Step>
  
  <Step title="Spawn container process">
    ```typescript
    const container = spawn(CONTAINER_RUNTIME_BIN, args, {
      stdio: ['ignore', 'pipe', 'pipe'],
    });
    ```
    
    Stdin is ignored — all IO goes through session databases. The entrypoint overrides to `bash -c 'exec bun run /app/src/index.ts'`.
  </Step>
  
  <Step title="Container runs independently">
    The agent-runner inside the container polls `inbound.db` for messages and writes results to `outbound.db`. The host's delivery polls pick up outbound messages. There is no stdin piping, stdout marker parsing, or IPC file watching.
  </Step>
  
  <Step title="Handle exit">
    When the container exits, it is removed from the active containers map and the session is marked as stopped. Stderr is logged at debug level during operation.
  </Step>
</Steps>

### Container arguments

From the `buildContainerArgs` function in `src/container-runner.ts`:

```typescript
async function buildContainerArgs(
  mounts: VolumeMount[],
  containerName: string,
  agentGroup: AgentGroup,
  containerConfig: ContainerConfig,
  provider: string,
  providerContribution: ProviderContainerContribution,
  agentIdentifier?: string,
): Promise<string[]> {
  const args: string[] = ['run', '--rm', '--name', containerName];

  // Only TZ and provider-contributed vars from env
  args.push('-e', `TZ=${TIMEZONE}`);

  // Provider-contributed env vars (e.g. XDG_DATA_HOME, NO_PROXY)
  if (providerContribution.env) {
    for (const [key, value] of Object.entries(providerContribution.env)) {
      args.push('-e', `${key}=${value}`);
    }
  }

  // OneCLI gateway — injects HTTPS_PROXY + certs for credential injection
  await onecli.ensureAgent({ name: agentGroup.name, identifier: agentIdentifier });
  const onecliApplied = await onecli.applyContainerConfig(args, {
    addHostMapping: false,
    agent: agentIdentifier,
  });

  // Host gateway, user mapping, volume mounts...
  args.push(...hostGatewayArgs());

  // Override entrypoint: run v2 entry point directly via Bun
  args.push('--entrypoint', 'bash');
  args.push(containerConfig.imageTag || CONTAINER_IMAGE);
  args.push('-c', 'exec bun run /app/src/index.ts');

  return args;
}
```

**Key flags:**
- `--rm` - Remove container after exit (ephemeral)
- `--name` - Unique container name (`nanoclaw-v2-{group}-{timestamp}`)
- `--add-host` - Maps `host.docker.internal` on Linux (automatic on macOS)
- `--user` - Run as host UID/GID for file permissions
- `--entrypoint bash` - Override default entrypoint to run Bun directly
- OneCLI SDK configures container networking for credential injection (HTTPS_PROXY + CA certs)

<Note>
Stdin is set to `ignore` (not piped). All configuration is read from `container.json` at `/workspace/agent/container.json` (mounted read-only). There are no credential env vars, no stdout markers, and no stdin JSON.
</Note>

### Volume mounts

All groups use the same mount structure. Mounts are built in `src/container-runner.ts`:

| Host path | Container path | Mode |
|-----------|---------------|------|
| Session directory | `/workspace` | Read-write |
| Agent group folder | `/workspace/agent` | Read-write |
| container.json | `/workspace/agent/container.json` | Read-only |
| Composed CLAUDE.md | `/workspace/agent/CLAUDE.md` | Read-only |
| Skill fragments | `/workspace/agent/.claude-fragments` | Read-only |
| Global memory | `/workspace/global` | Read-only |
| Shared CLAUDE.md | `/app/CLAUDE.md` | Read-only |
| Claude SDK state | `/home/node/.claude` | Read-write |
| Agent-runner source | `/app/src` | Read-only |
| Shared skills | `/app/skills` | Read-only |
| Additional mounts | `/workspace/extra/*` | Per-config |
| Provider mounts | Various | Per-provider |

<Tip>
Agent-runner source is shared read-only across all groups from `container/agent-runner/src/`. Source-only changes take effect on next container spawn without an image rebuild.
</Tip>

### Stale detection

There is no host-side wall-clock timeout in v2. Instead, stale containers are detected by the host sweep (`src/host-sweep.ts`), which runs every 60 seconds:

- **Heartbeat**: Container touches `/workspace/.heartbeat` periodically. Stale mtime indicates a stuck container.
- **Processing claims**: The `processing_ack` table tracks which messages the container is actively processing. Long-outstanding claims indicate stuck work.
- **Container state**: The `container_state` table records current tool-in-flight state for stuck detection.

When a stuck container is detected, the host kills it via `docker stop` and marks the session as stopped.

<Note>
This approach avoids killing containers doing legitimate long-running work. The agent-runner's own idle timeout causes it to exit gracefully when there are no pending messages — no host-side timer is needed.
</Note>

## Runtime management

### Ensuring runtime is available

On startup, NanoClaw verifies the container runtime is accessible via `ensureContainerRuntimeRunning` in `src/container-runtime.ts`:

```typescript
export function ensureContainerRuntimeRunning(): void {
  try {
    execSync(`${CONTAINER_RUNTIME_BIN} info`, {
      stdio: 'pipe',
      timeout: 10000,
    });
    logger.debug('Container runtime already running');
  } catch (err) {
    logger.error({ err }, 'Failed to reach container runtime');
    // Display error banner
    throw new Error('Container runtime is required but failed to start');
  }
}
```

### Cleaning up orphans

Orphaned containers from previous runs are cleaned up at startup via `cleanupOrphans` in `src/container-runtime.ts`:

```typescript
export function cleanupOrphans(): void {
  try {
    const output = execSync(
      `${CONTAINER_RUNTIME_BIN} ps --filter name=nanoclaw- --format '{{.Names}}'`,
      { stdio: ['pipe', 'pipe', 'pipe'], encoding: 'utf-8' },
    );
    const orphans = output.trim().split('\n').filter(Boolean);
    for (const name of orphans) {
      try {
        stopContainer(name);
      } catch {
        /* already stopped */
      }
    }
    if (orphans.length > 0) {
      logger.info(
        { count: orphans.length, names: orphans },
        'Stopped orphaned containers',
      );
    }
  } catch (err) {
    logger.warn({ err }, 'Failed to clean up orphaned containers');
  }
}
```

## IO model

All IO between host and container goes through per-session SQLite databases. There are no stdout markers, no stdin piping, and no IPC file watchers.

- **Inbound**: Host writes `messages_in` rows to `inbound.db`. Container polls for pending rows.
- **Outbound**: Container writes `messages_out` rows to `outbound.db`. Host delivery polls pick them up.
- **Liveness**: Container touches `/workspace/.heartbeat`. Host sweep checks mtime.
- **Seq parity**: Host uses even sequence numbers, container uses odd — this prevents sequence conflicts without locking.

Each database has exactly one writer, eliminating cross-mount lock contention. The `journal_mode=DELETE` pragma is required for cross-mount visibility between host and container.
